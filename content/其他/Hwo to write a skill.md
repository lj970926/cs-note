# 如何创建一个 Skill：内容怎么写 & 核心策略

> [!abstract] 这篇教程解决什么问题 不讲怎么点按钮、怎么打包安装，而是聚焦在一个 Skill 的**灵魂**：`SKILL.md` 里面的内容到底怎么写，以及背后有哪些反复奏效的策略。读完你应该能独立写出一个不会"过拟合"、能被可靠触发、并且让模型真正"听懂"的 Skill。

---

## 1. 先建立一个正确的心智模型

一个 Skill 本质上就是 **一个文件夹**，里面有一份必需的 `SKILL.md`，外加一些可选的资源（脚本、参考文档、模板）。

但比文件结构更重要的，是理解 Skill 的工作方式 —— 它不是一次性全部塞进模型脑子里的。它是**按需分层加载**的。把这个原理想透，后面所有的写法选择都会变得顺理成章。

> [!tip] 一句话抓住本质 Skill 不是"给模型的一篇说明书"，而是"在合适的时机、把合适的信息、喂给一个本来就很聪明的模型"的一套机制。你写的不是规则手册，而是一个**协作框架**。

---

## 2. Skill 的结构（Anatomy）

```
skill-name/
├── SKILL.md            (必需)
│   ├── YAML frontmatter  → name、description（必填）
│   └── Markdown 正文     → 给模型的指令
└── 可选资源
    ├── scripts/        → 可执行代码，处理确定性 / 重复性任务
    ├── references/     → 需要时才读入上下文的参考文档
    └── assets/         → 输出里会用到的文件（模板、图标、字体）
```

|资源类型|放什么|什么时候被加载|
|---|---|---|
|`scripts/`|脚本（如 `create_docx.py`）|需要时执行，**可以不读进上下文直接跑**|
|`references/`|大段参考资料、各种变体的细则|模型判断需要时才读|
|`assets/`|模板、字体、图标等成品素材|生成输出时使用|

---

## 3. 最核心的策略：渐进式披露（Progressive Disclosure）

这是写好 Skill 的**第一性原理**。整套加载分三层：

|层级|内容|加载时机|体量参考|
|---|---|---|---|
|1. 元数据|`name` + `description`|**永远在上下文里**|~100 词|
|2. 正文|`SKILL.md` 的 Markdown 主体|Skill 被触发时载入|理想 < 500 行|
|3. 资源|`scripts/` `references/` `assets/`|模型按需调用|不限量|

> [!important] 为什么这件事这么关键 第 1 层永远占用上下文，所以要极度精炼。第 2 层只在触发后才出现，可以详细一些但仍要克制。第 3 层近乎免费——脚本甚至能在不读入正文的情况下直接执行。**把信息放在正确的层级，是写 Skill 的核心功夫。**

实践要点：

- `SKILL.md` 正文尽量控制在 **500 行以内**。快到上限时，不要硬塞，而是**加一层目录结构**，并清楚地指引模型"接下来该去读哪个文件"。
- 引用 `references/` 里的文件时，要写清楚**什么情况下该去读它**。
- 参考文件超过 ~300 行，**在文件开头放一个目录（TOC）**。

---

## 4. 写 frontmatter：`name` 和 `description`

```yaml
---
name: skill-name
description: 这个 skill 做什么 + 什么时候该用它
---
```

### `name`

简短的标识符，作为文件夹名和调用名，保持稳定。

### `description` —— 整个 Skill 里最重要的一句话

> [!warning] 这是触发机制本身 `description` 是模型决定"要不要看这个 Skill"的**唯一依据**。它必须同时回答两件事：**这个 skill 做什么**，以及**在什么场景下该用它**。所有"何时使用"的信息都写在这里，**不要**放进正文。

一个反复出现的坑：模型倾向于**「欠触发」**——明明该用却没用。对策是把 description 写得**稍微"凶"一点 / 主动一点**，明确列出触发场景，哪怕用户没明说关键词。

> [!example] 对比一下 **太弱 👎** `How to build a simple dashboard to display internal data.`
> 
> **够主动 👍** `How to build a simple dashboard to display internal data. Make sure to use this skill whenever the user mentions dashboards, data visualization, internal metrics, or wants to display any kind of company data, even if they don't explicitly ask for a 'dashboard.'`

---

## 5. 正文怎么写：六条核心策略

这是你问的重点——`SKILL.md` 主体内容到底该怎么落笔。

### 策略一：解释「为什么」，而不是堆砌全大写的 MUST

这是最重要、也最反直觉的一条。今天的模型很聪明，有不错的"心智推理"能力。当你**解释清楚某件事为什么重要**，模型能举一反三；而当你只甩一堆 `ALWAYS` / `NEVER` 的硬规则，反而会把它框死。

> [!danger] 黄色警报 如果你发现自己在正文里写了一连串全大写的 `MUST` / `ALWAYS` / `NEVER`，或者套了非常死板的结构——停下来。这通常是个信号：你应该换种方式，**讲清背后的道理**，让模型理解为什么这样做才对。这比命令式约束更人性、更强大、也更有效。

### 策略二：用祈使句，并信任模型

指令优先用**祈使句**（"提取表格"，而不是"你应该提取表格"）。把模型当成一个聪明的协作者，给它一个好框架，让它发挥，而不是规定每一个微小动作。

### 策略三：定义清晰的输出格式

如果输出有固定结构，直接给模板：

```markdown
## Report structure
始终使用这个模板：
# [标题]
## 执行摘要
## 关键发现
## 建议
```

### 策略四：给例子（尤其是 Input → Output）

例子比抽象描述有用得多：

```markdown
## Commit message 格式
**示例 1：**
Input: Added user authentication with JWT tokens
Output: feat(auth): implement JWT-based authentication
```

### 策略五：不要过拟合，保持精简

> [!quote] 心法 你写的 Skill 会被使用**成千上万次**，面对各种各样你没见过的输入。如果它只对你测试时那几个例子有效，那它就是没用的。

- 遇到顽固的问题，别加越来越细的"补丁规则"。试着**换个比喻、换个工作模式**去描述，往往一下就通了。
- 定期用"新鲜的眼睛"重读草稿，**删掉不出力的内容**。每一句话都要为它的存在付租金。

### 策略六：该用脚本时，就用脚本

如果某类任务是**确定性的、重复的**（比如生成 docx、画固定样式的图表），与其每次让模型现写代码，不如**把脚本写好放进 `scripts/`**，在正文里告诉它去调用。这样每一次调用都省去了"重新造轮子"的时间。

> [!tip] 一个识别信号 如果你在测试时发现，多个测试用例里模型都**各自独立写了一份几乎一样的辅助脚本**，那就是在告诉你：这段逻辑该被沉淀成一个 bundled script 了。

---

## 6. 进阶：用 `references/` 按变体拆分

当一个 Skill 要支持**多个领域 / 框架**时，按变体组织参考文件，让模型只读它当下需要的那一个：

```
cloud-deploy/
├── SKILL.md          (主流程 + 怎么选择)
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```

`SKILL.md` 负责"判断与路由"，具体细则下沉到 `references/`。这正是策略三（渐进式披露）在多领域场景下的落地。

---

## 7. 不出意外原则（Principle of Lack of Surprise）

> [!warning] 底线 Skill 的实际行为，必须和它的描述一致——**不能给用户意外**。不要包含恶意代码，不要做"描述成 A、实际干 B"的事，不要协助未授权访问或数据外泄。（"扮演某个角色"这类是没问题的。）

---

## 8. 写完之后：测试与迭代循环

内容写好只是开始。真正打磨一个 Skill 靠的是一个循环：

> **草稿 → 跑测试用例 → 评估结果 → 改进 → 重复**

- 写完草稿后，构造 **2~3 个真实用户会说的测试 prompt**（不是抽象的"格式化这个数据"，而是带文件名、列名、背景的具体请求）。
- 用带 Skill 和不带 Skill 各跑一遍，**对比差异**。
- 重点看**过程**而不只是最终输出：如果 Skill 让模型绕了很多无用的弯路，就把导致这种行为的部分删掉再试。
- 一直迭代到你满意、或者反馈都没意见、或者不再有明显进步为止。

> [!note] 关于 description 的二次优化 整个 Skill 稳定后，可以专门再优化一次 `description`：构造一批"该触发 / 不该触发"的查询（**最有价值的是"近似但不该触发"的边缘案例**），反复测它的触发准确率。

---

## 9. 落笔前的检查清单

- [ ] `description` 同时说清了**做什么** + **何时用**，并且足够主动（防欠触发）
- [ ] 所有"何时使用"的信息都在 description 里，没散落在正文
- [ ] 正文 < 500 行；超了就分层 + 加指引
- [ ] 信息放在了正确的加载层级（渐进式披露）
- [ ] 用"为什么"代替了成串的全大写 MUST
- [ ] 给了输出模板和 Input→Output 例子
- [ ] 删掉了不出力的句子，没有过拟合到测试样例
- [ ] 重复性确定任务沉淀成了 `scripts/`
- [ ] 大参考文件（>300 行）开头有目录
- [ ] 行为和描述一致，没有意外

```markdown
---
name: my-skill
description: 做什么 + 何时用（写得主动一点，列出触发场景）。
---

# My Skill

一句话说明这个 skill 帮模型完成什么。

## 工作流程
1. 第一步（祈使句）
2. 第二步……
（解释关键步骤为什么这么做）

## 输出格式
始终使用这个模板：
...

## 示例
**示例 1：**
Input: ...
Output: ...

## 需要时参考
- 处理 X 场景时，读 `references/x.md`
- 生成 Y 时，运行 `scripts/build_y.py`
```
